# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Added

- Schéma d'identité & RBAC (tâche 1.1) : tables `user`, `space`, `space_membership`, `role`, `permission`, `role_permission` (`server/db/schema/identity.ts`), PK en UUID (`crypto.randomUUID()` via `$defaultFn`), `created_at`/`updated_at` (rafraîchi automatiquement à chaque écriture), `role.is_system`, unicité `(space_id, role.name)` et `(user_id, space_id)` sur `space_membership`.
- Auth : sessions + hash de mot de passe (tâche 1.2) : module `nuxt-auth-utils` branché dans `nuxt.config.ts`, `NUXT_SESSION_PASSWORD` (scellement des cookies de session) et `NUXT_APP_SECRET` (chiffrement des secrets connecteurs, phase 4) documentés dans `.env.example`. Le hachage utilise directement `hashPassword()` / `verifyPassword()` fournis par `nuxt-auth-utils` (scrypt via `@adonisjs/hash`, format PHC + `passwordNeedsReHash()`) — pas de wrapper maison.
- Test e2e API (`@nuxt/test-utils`) sur les routes de session exposées par `nuxt-auth-utils` (`server/utils/__tests__/auth-session.e2e.test.ts`) : `GET /api/_auth/session` sans cookie, `DELETE /api/_auth/session` (déconnexion + purge du cookie).
- Premier lancement (setup admin) + seed rôles (tâche 1.3) : `server/utils/roles.ts` (catalogue de 8 permissions MVP + `ensureSystemRoles()` qui crée les rôles système `Owner`/`Member` d'un espace et leur attribue toutes les permissions), `server/api/setup.get.ts` (`{ needsSetup }`), `server/api/setup.post.ts` (validation Zod, transaction Drizzle créant l'espace + les rôles + le premier utilisateur + le rattachement Owner, puis connexion via `setUserSession`), `app/pages/setup.vue` (formulaire), `app/middleware/setup-check.global.ts` (redirige toute navigation vers `/setup` tant qu'aucun utilisateur n'existe, et inversement). Ajout de la dépendance `zod` (prévue par la stack, absente jusqu'ici) et de `shared/types/auth.d.ts` (typage `User` de `#auth-utils`).
- Login / logout (tâche 1.4) : `server/api/auth/login.post.ts` (validation Zod, message générique « Identifiants invalides » sur email inconnu ou mot de passe incorrect, vérification du mot de passe systématiquement effectuée — y compris avec un hash factice quand l'email n'existe pas — pour ne pas laisser fuiter l'existence d'un compte par le temps de réponse, session posée via `setUserSession` avec `rememberMe` optionnel), `server/api/auth/logout.post.ts` (`clearUserSession`), `server/api/auth/me.get.ts`, `app/pages/auth/login.vue` (formulaire avec affichage/masquage du mot de passe). `app/middleware/01-login-check.global.ts` redirige toute route non authentifiée vers `/auth/login` (hors `/setup`, pour ne pas entrer en boucle avec le contrôle de premier lancement) et renvoie vers `/` si un utilisateur déjà connecté visite `/auth/login` ; `server/middleware/session-refresh.ts` prolonge la session de façon glissante (30 jours, constante partagée `server/utils/session.ts`) tant que `rememberMe` est actif. Déconnexion câblée dans `app/components/layout/Navbar.vue` via `useUserSession().clear()` (et non un simple appel `$fetch`, pour que l'état de session côté client soit invalidé en même temps que le cookie).
- Middleware d'auth + scoping espace (tâche 1.5) : `server/utils/context.ts` (`resolveRequestContext()`) résout `{ userId, spaceId, projectId }` à partir de la session — sélectionne l'unique/première `space_membership` de l'utilisateur par défaut, ou valide un `spaceId` explicite passé en paramètre — et rejette (403) l'accès à un espace dont l'utilisateur n'est pas membre (`projectId` reste `null` tant que la table `project` de la tâche 1.8 n'existe pas). Branché sur `GET /api/auth/me`, qui accepte désormais un `?spaceId=` optionnel et renvoie `spaceId`/`projectId` en plus du `user`. Couvert par `server/utils/__tests__/context.e2e.test.ts` (session absente, espace dont l'utilisateur est membre, rejet sur un espace étranger).
- Logging structuré avec `pino` (`server/utils/logger.ts`) : sortie JSON par défaut, ré-écrite en clair/colorée via `pino-pretty` uniquement en dev local (`import.meta.dev`), pour ne jamais charger `pino-pretty` (devDependency) en production. Niveau `debug` en dev / `info` en prod (surchageable via `NUXT_LOG_LEVEL`). `server/plugins/logger.ts` logge chaque requête (méthode, chemin, code, durée) via les hooks Nitro `request`/`afterResponse` (détail de la requête — query, user-agent — en `debug`), et route les erreurs 5xx/non gérées vers le hook `error` (les erreurs 4xx volontaires, ex. identifiants invalides, ne sont pas re-loggées ici : elles le sont déjà, plus précisément, au niveau du handler). Requêtes SQL Drizzle tracées en `debug` (`server/utils/db.ts`). Événements d'auth significatifs (login réussi/refusé, logout, accès à un espace refusé, premier lancement) loggés en `info`/`warn` — donc visibles aussi en production, pas seulement en debug local.
- Espaces : CRUD + sélecteur (tâche 1.6) : `GET /api/spaces` (liste les espaces dont l'utilisateur est membre + l'espace actif résolu via `resolveRequestContext()`), `POST /api/spaces` (crée un espace, l'utilisateur courant en devient Owner via `ensureSystemRoles()`, et y bascule), `POST /api/spaces/[id]/select` (change l'espace actif, 403 si l'utilisateur n'est pas membre — délégué à `resolveRequestContext()`). L'espace actif est mémorisé dans la session (`UserSession.currentSpaceId`, ajouté à `shared/types/auth.d.ts`), lu en priorité par `resolveRequestContext()`. Côté front, `app/composables/useCurrentSpace.ts` (partagé entre composants via la clé `useAsyncData("spaces")`, sans Pinia) expose l'espace actif et pilote `app/app.vue` : le `data-theme` sur `<html>` reflète désormais le thème de l'espace actif (au lieu d'être codé en dur à `"light"`), y compris au premier rendu SSR (nécessite `useRequestFetch()` plutôt que le `$fetch` global, qui ne relaie pas les cookies de la requête entrante lors d'un appel serveur→serveur pendant le SSR). `app/components/layout/SpaceSwitcher.vue` (`<select>` natif dans la sidebar + modale de création téléportée à la racine du `<body>` pour éviter tout effet de clipping/animation parasite dans le drawer) liste les espaces, permet d'en changer et d'en créer un. Couvert par `server/api/spaces/__tests__/spaces.e2e.test.ts`.
- Schéma projet (tâche 1.8) : table `project` (`server/db/schema/project.ts`) rattachée à l'espace (`space_id` avec `ON DELETE CASCADE`), colonnes `name`, `theme` (nullable — `null` = hérite du thème de l'espace, la surcharge sera appliquée en tâche 1.9) et `deliverable_label` (défaut `"Livrable"`, libellé du concept « livrable » propre au projet), migration `0001_project.sql`. Les helpers de schéma `id()` (PK UUID) et `timestamps` (`created_at`/`updated_at`), jusqu'ici locaux à `identity.ts`, sont extraits dans `server/db/schema/helpers.ts` pour être partagés par les prochains fichiers de schéma (couches canonique, augmentation, historisation).
- Projets : CRUD + sélecteur + thème (tâche 1.9) : `GET /api/projects` (liste les projets de l'espace actif + le projet actif résolu par `resolveRequestContext()`), `POST /api/projects` (crée un projet dans l'espace actif — `name`, `theme` optionnel, `deliverableLabel` optionnel — et y bascule), `POST /api/projects/[id]/select` (change le projet actif, 403 si le projet n'appartient pas à l'espace actif — délégué à `resolveRequestContext()`). Le projet actif est mémorisé dans la session (`UserSession.currentProjectId`, ajouté à `shared/types/auth.d.ts`). Côté front, `app/composables/useCurrentProject.ts` (partagé via la clé `useAsyncData("projects")`, relit la liste quand l'espace actif change) expose le projet actif ; `app/components/layout/ProjectSwitcher.vue` (dropdown verre dans la sidebar sous le sélecteur d'espace + modale de création avec choix du thème) liste les projets, permet d'en changer et d'en créer un. Le `data-theme` sur `<html>` (`app/app.vue`) applique désormais la surcharge : thème du projet actif, sinon thème de l'espace, sinon `oneeo`. Couvert par `server/api/projects/__tests__/projects.e2e.test.ts`.
- Utilisateurs de l'espace : création par l'Owner (tâche 1.7) : `server/utils/roles.ts` gagne `isSpaceOwner()` (vérifie le rôle de l'utilisateur pour un espace donné). `GET /api/spaces/[id]/members` (liste les membres, accessible à tout membre de l'espace) et `POST /api/spaces/[id]/members` (réservé à l'Owner — 403 sinon ; crée l'utilisateur avec un mot de passe temporaire généré côté serveur, rôle Member fixe, `mustChangePassword: true` ; 409 si l'email existe déjà). Le changement de mot de passe est réellement forcé à la première connexion : `User.mustChangePassword` ajouté à `shared/types/auth.d.ts` et à la session posée par `login.post.ts`/`setup.post.ts`, `server/api/auth/set-password.post.ts` (change le mot de passe de l'utilisateur courant sans redemander l'ancien — il vient de se connecter avec le mot de passe temporaire), page `app/pages/changer-mot-de-passe.vue`, et deux branches supplémentaires dans `01-login-check.global.ts` qui redirigent vers cette page tant que le flag est actif (et en repartent une fois le mot de passe changé). Interface de gestion sur `app/pages/equipes.vue` (table des membres + rôle + statut, bouton « Ajouter un utilisateur » en haut, modale affichant le mot de passe temporaire généré une seule fois — pas d'envoi d'email en MVP). Couvert par `server/api/spaces/__tests__/members.e2e.test.ts` (création, doublon d'email, 403 non-Owner, 403 espace étranger, connexion avec le mot de passe temporaire puis changement effectif).

### Changed

- `resolveRequestContext()` (`server/utils/context.ts`) résout désormais le projet actif (tâche 1.9) : cadré à l'espace résolu, il lit un `projectId` explicite (repli sur celui mémorisé en session, puis sur le premier projet de l'espace) et rejette (403) un projet appartenant à un autre espace. `projectId` reste `null` pour un espace sans projet.
- `useDb()` active `PRAGMA foreign_keys = ON` afin que les contraintes `ON DELETE CASCADE`/`RESTRICT` du schéma soient appliquées par SQLite.
- `vitest.config.ts` : `testTimeout`/`hookTimeout` portés à 30 s pour accommoder le démarrage d'un vrai serveur Nuxt dans les tests `*.e2e.test.ts`.
- `npm run test` applique désormais les migrations (`pretest`) avant de lancer Vitest, pour que les tests e2e qui démarrent un vrai serveur Nuxt (middleware/API dépendant du schéma) n'échouent pas sur une base vierge non migrée.
- `app/middleware/setup-check.global.ts` renommé en `00-setup-check.global.ts` (préfixe numérique) afin de garantir qu'il s'exécute avant `01-login-check.global.ts` — l'ordre alphabétique des fichiers de middleware global détermine leur ordre d'exécution dans Nuxt.
- Route de connexion renommée de `/auth/login` vers `/login` (tâche 1.5) : `app/pages/auth/login.vue` → `app/pages/login.vue`, redirections mises à jour dans `01-login-check.global.ts` et `Navbar.vue`.
- `vitest.config.ts` : `fileParallelism: false` — un seul fichier SQLite (`data/oneeo.sqlite`, pas de base de test isolée) et un seul verrou de dev server Nuxt, donc deux fichiers `*.e2e.test.ts` lancés en parallèle se marchaient dessus.

### Removed

- Table de test `health_check` (tâche 0.4), remplacée par le schéma métier de la tâche 1.1.

## [0.0.1] - 2026-07-07

### Added

- Initialisation du projet Nuxt 4 (tâche 0.1) : `compatibilityDate` fixée au 2025-07-15, TypeScript strict, version Nuxt 4.4.8 épinglée, lockfile committé.
- Intégration Tailwind CSS 4 + DaisyUI 5 (tâche 0.2) : config en CSS via `@tailwindcss/vite`, tous les thèmes DaisyUI activés, `data-theme` posé sur `<html>` via `useHead()`.
- Ajout Biome 2.5.2 (tâche 0.3) : lint, format `--write`, imports triés, quotes doubles, support directives Tailwind CSS.
- Intégration Drizzle ORM + better-sqlite3 (tâche 0.4) : `useDb()` typé, config `drizzle-kit`, scripts `db:generate` / `db:migrate` / `db:seed`, table de test migrée sur base vierge.
- Console interactive Drizzle façon `rails console` (`npm run console`) : `db` et les tables du schéma disponibles directement dans un REPL Node avec top-level await.
- Structure de dossiers hexagonale côté serveur (tâche 0.5) : `server/domain/` (types + calculs purs), `server/application/` (services orchestrateurs), `server/adapters/{jira,manual,llm,mail}/` (implémentations des ports), `server/api/` (endpoints Nitro) ; `server/README.md` documente le rôle de chaque dossier.
- Intégration Vitest (tâche 0.6) : `vitest.config.ts` ciblant `server/**/__tests__/**/*.test.ts`, script `npm run test`, test smoke sur une fonction pure (`server/domain/__tests__/smoke.test.ts`) exécuté sans runtime Nuxt.
- Layout de base navigable (tâche 0.7) : drawer DaisyUI sidebar always-open sur desktop / hamburger mobile, navbar sticky avec notifications et menu utilisateur, 8 pages placeholder (Dashboard, Sprints, Livrables, Équipes, Temps & Congés, Anomalies & SLA, Rapports & IA, Paramètres).
- Intégration HugeIcons (`@hugeicons/core-free-icons` + `@hugeicons/vue`) pour les icônes de navigation et de la navbar.
- Wrapper ApexCharts + helper de thème (tâche 0.8) : `app/components/charts/BaseChart.client.vue` (composant client-only avec démo intégrée), `useChartTheme()` qui lit les variables CSS DaisyUI (`--color-primary`, `--color-base-content`, etc.) et les injecte dans les options ApexCharts, avec réactivité au changement de `data-theme` via `MutationObserver`.

### Changed

- Remplacement de Biome par ESLint (`@nuxt/eslint`) + Prettier : résout les faux positifs "unused import" sur les templates Vue, config Prettier double quotes / 2 espaces, règle `no-explicit-any` désactivée.
- Toolbar ApexCharts masquée par défaut dans `BaseChart.client.vue` (surchageable via les options).
